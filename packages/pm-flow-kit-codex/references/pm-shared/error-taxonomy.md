# PM Flow Kit Error Taxonomy

Use this shared taxonomy to prevent uncontrolled output. Each skill should identify whether an error is a recoverable gap, a handoff, or a blocking issue.

## Severity

| Severity | Meaning | Required action |
|---|---|---|
| E0 | Minor wording or formatting issue | Record and continue |
| E1 | Missing detail that can be handled with an assumption | Record assumption and continue |
| E2 | Missing or conflicting input that affects output quality | Ask one focused question or create a blocking open question |
| E3 | Incorrect source, unsafe assumption, or artifact mismatch | Block downstream output until corrected or explicitly accepted |
| E4 | Legal, security, financial, compliance, safety, or production-risk error | Stop and require owner decision |

## Context Errors

- Knowledge base is missing, but the agent claims it used it.
- Agent reads the whole knowledge base instead of the manifest or scoped sources.
- Source material conflicts and the conflict is hidden.
- Archived, outdated, or unowned material is treated as authoritative.
- Company or project context leaks across unrelated workspaces.

## Requirement Errors

- A solution request is treated as the final requirement without problem analysis.
- A solution, UI control, or technical stack is written as the problem statement.
- Target users, roles, client surfaces, trigger, or expected outcome are missing.
- Scope and non-scope are not separated.
- Assumptions are written as confirmed facts.
- Source errors are not classified before downstream artifacts are produced.

## Workflow Control Errors

- A fixed full workflow is run for every request regardless of size, risk, or artifact plan.
- Technical stack selection is forced before product boundary, scenarios, fields, and constraints are clear.
- PRD, diagram, prototype, export, and review artifacts are generated as a bundle without user need or routing decision.
- A large external reference pack is copied into the skill instead of extracting reusable rules or failure cases.

## Metric Errors

- Feature launch is treated as the success metric.
- Page views or exposure counts are treated as activation without proving the value moment.
- Metrics have no formula, source, baseline, owner, or decision threshold.
- Guardrail metrics are missing for risky requirements.
- Baselines, events, or dashboards are invented.
- Metric changes are interpreted as feature impact without data-quality, cohort, channel, version, segment, or seasonality checks.
- Metrics are disconnected from scenarios, use cases, or acceptance criteria.

## Scenario And Use Case Errors

- Scenarios describe broad phases instead of role + client surface + trigger + user action + system response.
- Use cases omit preconditions, main flow, exception flow, post-condition, or acceptance lens.
- Fields, permissions, and data sources cannot be traced to scenarios.
- Exception flows are summarized as "handle errors" without concrete system behavior.

## Artifact Planning Errors

- Every artifact is generated by default.
- Prototype review is generated even when no review artifact is needed.
- Diagrams are embedded in the PRD when the workspace requires standalone diagram files.
- Workspace directory standards are ignored without explanation.

## PRD Errors

- Every requirement is forced into a long PRD.
- PRD level is not explicit.
- Field definitions are buried in prose.
- Only happy-path behavior is documented.
- Subjective acceptance criteria such as "easy to use" or "beautiful" are used without observable tests.
- Requirement granularity is inconsistent across same-level features.
- Technical constraints, permission boundaries, or integration dependencies are ignored when they affect feasibility.
- Acceptance criteria cannot be turned into test scenarios.
- Metrics, scenarios, and use cases are not referenced.

## Prototype Errors

- Prototype pretends to be production code.
- Prototype depends on CDN resources when offline output is required.
- Design guidelines, component library, frontend framework, or font constraints are ignored.
- Mobile or H5 prototypes omit loading, empty, error, permission, timeout, or duplicate-action states when scenarios require them.
- UI fields cannot be traced to the PRD or field dictionary.

## Diagram Errors

- One diagram mixes product flow, data model, architecture, roadmap, and delivery plan without a single purpose.
- Swimlane diagrams use vague departments instead of real actors, systems, teams, or client surfaces.
- Diagram type is selected by habit instead of purpose: flow, state, sequence, architecture, roadmap, or concept map.
- Mermaid output is produced without a standalone file path when the workspace requires standalone diagrams.

## Review Errors

- Review artifact is treated as mandatory.
- Findings are vague or not mapped to rules, fields, acceptance criteria, or open questions.
- Review conclusions silently change scope or requirements.

## Humanizer Errors

- Polishing changes facts, scope, metrics, acceptance criteria, or decisions.
- Product documentation is turned into marketing copy.
- Open questions or assumptions are rewritten as certainty.
