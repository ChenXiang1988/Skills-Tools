# pm-flow-kit 插件介绍

## 一句话介绍

`pm-flow-kit` 是一套面向产品经理和产品型 Agent 的需求交付插件。它让 Agent 按"调研 → 边界 → 原型 → PRD → 评审"的顺序推进产品工作，并把工作区、知识库边界、产物、决策、上下文和门禁（阶段关闭前的检查项）都保留下来。

## 它解决什么问题

很多 AI 工具都能写 PRD、画原型、总结需求，但常见问题就这几个：

- 跳步骤：还没澄清边界，PRD 已经写完了。
- 把假设当事实。
- 不知道什么时候该查知识库，什么时候该记下"此处不适用"。
- 写出来的 PRD 看着完整，却没法评审、没法验收、没法交付。
- 原型、PRD、流程图、评审各自为政，对不上。

`pm-flow-kit` 要解决的，就是"AI 能写东西，但不一定按产品流程交付"这件事。它把产品工作拆成可追踪的阶段，让 Agent 在每一步都清楚：现在该做什么、要产出什么、引用哪份规范、什么时候能关闭这个阶段。

## 适合谁用

- **产品经理**：把模糊需求推进成可评审、可交付的方案。
- **业务负责人**：让需求过程透明，少做口头确认，少返工。
- **设计 / 研发 / 测试**：拿到边界清楚、规则明确、验收能落地的材料。
- **用 AI Agent 的团队**：让 Agent 不只是吐文本，而是按产品工作流协作。

## 核心能力

| 能力 | 对应 skill | 说明 |
|---|---|---|
| 流程总控 | `pm-flow` | 管理阶段、状态、产物、决策和门禁 |
| 上下文约束 | `pm-context-contract` | 管理知识库、领域规范、输入材料与冲突优先级 |
| HTML 原型 | `pm-html-prototype` | 生成离线 HTML 原型、设备框、设计规范引用和组件库映射 |
| 原型评审 | `pm-prototype-review` | 生成原型评审件、字段规则联动和评审材料；只在需要评审件时使用 |
| PRD 写作 | `pm-prd` | 按需求量级输出 L1 / L2 / L3 PRD |
| 指标定义 | `pm-metrics-definition` | 定义业务指标、用户行为指标、输入指标、护栏指标、基线、数据源和判断阈值 |
| 图表生成 | `pm-mermaid-diagram` | 画流程图、泳道图、状态机、时序图等 Mermaid 图表 |
| 文档润色 | `humanizer` | 去掉 PRD 的 AI 味，只改表达，不动事实 |

## 工作流程

默认把一个需求拆成五个阶段：

```
research → boundary → prototype → prd → review
```

不同需求走不同路径：

| 类型 | 适用场景 | 阶段 |
|---|---|---|
| `new-platform` | 新平台、新业务、新系统 | research → boundary → prototype → prd → review |
| `iteration` | 常规产品迭代 | boundary → prototype → prd → review |
| `micro` | 小改动、轻量需求 | prd → review |

每个阶段都要求有产物和检查项。不能只说"做完了"，得看得到文件、决策和门禁确认。

## 典型使用场景

### 1. 从一句话需求到 PRD

用户只丢一句话过来，比如：

> 我们想做一个退款审批流程。

`pm-flow-kit` 会先让 Agent 把目标用户、业务场景、范围、非范围、关键状态和验收标准问清楚，而不是直接甩一份大 PRD 出来。

### 2. 引用知识库和领域规范

项目里只要有知识库、术语表、业务规则、设计规范或组件库，`pm-flow` 会先确认知识库路径或确认"不使用外部知识库"，再由 `pm-context-contract` 定好上下文读取规则。Agent 只读当前阶段相关的资料，不会把整个知识库一股脑塞进上下文。

### 3. 生成 HTML 原型

`pm-html-prototype` 生成的原型可以离线打开，支持：

- PC / 手机 / PDA 视图
- 钉钉进步体字体
- 界面字段和右侧规则文档通过 `data-link` 联动
- 本地打开，不依赖 CDN

知识库里有设计规范或组件库就优先引用；没有，就用内置基础规范自闭环，同时把设计假设记下来。

### 4. 输出合适量级的 PRD

`pm-prd` 不先套厚模板，而是先判断需求量级：

- **L1**：轻量 PRD，适合小改动。
- **L2**：标准 PRD，适合常规迭代。
- **L3**：深度 PRD，适合复杂业务、复杂状态、跨角色或跨系统需求。

### 5. 画流程图和泳道图

`pm-mermaid-diagram` 支持流程图、状态机、时序图和泳道图。凡是涉及"谁负责哪一步""跨部门怎么协作""角色和系统如何交接"的需求，优先出泳道图。

### 6. PRD 去 AI 味

PRD 内容定下来之后，用 `humanizer` 做最后一层表达润色。它只改表达，不允许动这些：

- 需求事实
- 范围和非范围
- 业务规则
- 字段含义
- 验收标准
- 决策和待确认事项

## 和普通 PRD 工具的区别

普通 PRD 工具盯着"生成一份文档"。`pm-flow-kit` 盯着"把需求按工作流推进到能评审、能交付"。差别在这几点：

- 有阶段状态，不会跳步。
- 有门禁检查，不能口头"完成"。
- 能先确认知识库边界，再引用知识库，不乱用上下文。
- 能同时产出原型、PRD、图表和评审材料。
- 能按需求量级选文档深度。
- 能记录假设和"不适用"的原因。

## 使用示例

### 启动一个常规迭代

```
使用 pm-flow-kit，启动一个 iteration 类型的产品需求流程。

项目名：退款审批优化
需求名：新增退款审批工作台

请先判断当前阶段，按 pm-flow 的流程推进，不要直接写 PRD。
如果需要上下文，请通过 pm-context-contract 读取 manifest。
```

### 生成 HTML 原型

```
使用 pm-prototype-review 生成一个离线 HTML 原型。

场景：退款审批工作台
角色：客服、财务、系统
设备：PC
页面：申请列表、申请详情、审核弹窗、驳回原因弹窗

如果知识库里有设计规范和组件库，请先通过 pm-context-contract 引用。
如果没有，请使用内置基础规范自闭环，并记录设计假设。
```

### 生成 PRD

```
使用 pm-prd 输出 PRD。

请先判断 PRD 等级：L1 / L2 / L3。
再选择对应模板。
如果涉及跨角色流程，请用 pm-mermaid-diagram 输出泳道图。
PRD 内容确认后，再用 humanizer 去 AI 味，但不能改变需求事实。
```

## 使用边界

`pm-flow-kit` 不是万能产品经理，也替代不了团队决策。它适合帮团队做这些事：

- 梳理需求
- 管理流程
- 产出文档
- 生成原型
- 引用规范
- 做阶段检查

它不该被用来：

- 绕过业务负责人确认
- 替代真实用户研究
- 把假设当事实
- 没依据时编造规范
- 为了视觉效果改业务规则

## 推荐介绍语

可以这样跟别人介绍：

> `pm-flow-kit` 是一套产品需求交付插件。它让 AI Agent 不只是写 PRD，而是按产品经理的工作流推进需求：先澄清边界，再做原型，再写 PRD，最后评审。它能引用团队知识库和设计规范，也能在没有规范时自闭环生成原型，并把假设记下来。

短一点的版本：

> `pm-flow-kit` 是一个让 Agent 按产品流程交付需求的插件，覆盖需求澄清、HTML 原型、PRD、流程图、评审和文档润色。
